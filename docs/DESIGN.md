# 最高気温マニア専用データベース＆ビジュアライザー 設計書

- ステータス: ドラフト v0.2
- 作成日: 2026-07-22
- 対象読者: 開発者（自分自身）、将来のコントリビューター

> **v0.2 の確定事項（2026-07-22）**: ホスティングを **GitHub Pages + GitHub Actions**、データ配信を **SQLite（`sql.js-httpvfs`）を主軸＋定番の重い集計のみ事前計算JSON** とする方針に確定。これに伴い、ランタイムのPostgreSQLサーバーは持たない（Postgresは廃案）。「歴代猛暑日回数ランキング」等のニッチな集計をパイプライン改修なしにフロントのSQLで無限に追加できることを重視した判断（4.2(d)・5.3・6章参照）。

## 1. コンセプト / 対象ユーザー

### 1.1 コンセプト

日本全国の**日別・最高気温**の記録に特化したデータベース兼ビジュアライザー。気象庁（JMA）の公式観測データのみを扱い、単なる数値ダッシュボードではなく、**「その日がどれだけ暑かったか」を視覚・体感として伝えるUI**を目指す。

猛暑日・酷暑日といった過酷な気温帯に入るほど、UI自体が「灼ける」ように演出がエスカレーションする。ユーザーが「うわ、これは酷暑日だ」と数字を読む前に画面から感じ取れることを目標にする。

### 1.2 対象ユーザー

- **最高気温マニア**: 「観測史上1位はどこか」「自分の住む町の最高記録は」といった問いを日常的に持つ人。
- **天候オタク／気象ファン**: 猛暑・酷暑のニュースが出るたびに過去記録と比較したい人。
- **データ好き／地理好き**: 地図とランキングを行き来してパターンを発見したい人。
- **想定外に配慮するユーザー層**: 研究者や自治体職員が概況を掴む用途にも耐えうる正確性は意識するが、一次情報源としての利用は想定しない（非スコープ参照）。

### 1.3 プロダクトの核となる体験

1. 「今日は全国のどこが一番暑かったか」を一目で見られる。
2. 「歴代最高気温トップ100」を眺めて記録の重みを体感できる。
3. 特定地点（例：熊谷、江川崎）のページで、その地点の全歴史を振り返れる。
4. 特定の日付を選ぶと、その日の全国分布と地図が同時に見られる。
5. 猛暑日・酷暑日のページ／カードには、気温に応じた色・揺らぎ・パルスなどの演出がかかる。

## 2. スコープ & 非スコープ

### 2.1 スコープ（v1で対応する）

- 気象庁が公開する**日本国内**の観測地点（気象官署・アメダス）の**日別最高気温**データ。
- 観測地点マスタ（緯度経度・種別・観測開始日を含む）。
- 3軸ランキング（全国歴代／地点別／日付別）。
- 日付別の全国分布ビュー（地図＋ランキング表）。
- 地点ページ（個別地点の全期間の推移・自己ベスト）。
- 気温連動のビジュアルエフェクト（夏日〜酷暑日の4段階演出）。
- 出典（気象庁）の明記とライセンス順守。

### 2.2 非スコープ（v1では対応しない）

- 海外（日本国外）の気温データ。
- 最低気温・平均気温・降水量・湿度など、最高気温以外の気象要素（将来検討の余地はあるが本設計書の対象外）。
- リアルタイム（速報値・分単位）観測データの逐次配信。日次バッチ更新を前提とする。
- 気象予測・予報機能。あくまで**過去の観測記録**を扱うプロダクトであり、天気予報アプリではない。
- ユーザー投稿・SNS的機能（コメント、フォロー等）。
- 気象庁データの再配布API化・商用二次利用の提供（自サイト内での表示・可視化のみ）。
- 多言語対応（日本語UIのみ。将来英語化は検討課題として9章に記載）。

## 3. 気温カテゴリ定義（表）

気象庁が定める公式の気温区分に準拠する。**酷暑日（40℃以上）は2025年より気象庁が正式に採用した公式用語**であり、非公式の俗称ではない。本プロダクトはこの4区分を軸にUI演出を設計する。

| カテゴリ | 定義（最高気温） | 気象庁公式用語 | 演出テーマカラー | 想定エフェクト |
|---|---|---|---|---|
| 夏日 (natsubi) | 25℃ 以上 | ○（公式） | 黄色系 (#FFD54F 相当) | 背景に薄い暖色グラデーション |
| 真夏日 (manatsubi) | 30℃ 以上 | ○（公式） | 橙色系 (#FF9800 相当) | グラデーション濃色化、アイコン変化 |
| 猛暑日 (mousho-bi) | 35℃ 以上 | ○（公式） | 赤色系 (#F4511E 相当) | 陽炎（heat-haze）シェーダー適用開始 |
| 酷暑日 (kokusho-bi) | 40℃ 以上 | ○（公式・2025年導入） | 深紅〜マゼンタ系 (#B71C1C〜#880E4F) | 陽炎強化＋画面全体の微振動（screen shake）＋数値のパルス点滅 |

補足:
- 各区分は下位区分を包含する（例: 36℃は「猛暑日」であり同時に「真夏日」「夏日」でもある）。UI上はその日の**最高到達区分**のみをバッジとして強調表示し、内部データとしては閾値との比較で全区分フラグを保持する。
- 区分の閾値は「以上」（境界値を含む）。35.0℃ちょうどは猛暑日。
- エフェクトの詳細な実装（シェーダー、パラメータ）は4.5節および8章 Phase4 で扱う。

## 4. 機能要件

### 4.1 気温DB（観測地点 × 日付 × 最高気温）

- 中核データは「観測地点」×「日付」×「その日の最高気温」の組。
- 検索軸:
  - **日付から**: 特定日、または期間（例: 2023年8月1日〜8月31日）を指定して該当レコードを取得。
  - **地点から**: 都道府県・地点名（かな検索対応）・地点種別（官署／アメダス）で絞り込み。
  - **気温帯から**: 「35℃以上のみ」「40℃以上のみ」のようにカテゴリ（3章）でフィルタ。
- 検索結果は常に「地点名・都道府県・日付・最高気温・カテゴリバッジ」を含む一覧として表示し、各行から地点ページ／日付詳細ページへ遷移できる。
- 将来的な全文検索（地点名のあいまい検索、旧地点名への対応）はPhase1以降で検討。

### 4.2 ランキング3軸

同一地点が複数のランキングエントリを持つのは自然な仕様とする（地点＋日付の組み合わせ単位でランキングするため、例えば熊谷が歴代トップ10に複数日エントリで登場してよい）。各エントリはクリックで該当日の詳細ページ（4.3節）へ遷移する。

#### (a) 歴代全国ランキング
- 全観測地点・全期間を通じた「最高気温トップN」。
- 例: 1位 41.1℃ 浜松（2020-08-17）、2位 41.1℃ 熊谷（2018-07-23）… のように、同着（タイ記録）も正しく扱う（同一気温は同順位表示、次の順位は人数分スキップ）。
- フィルタ: 期間指定、都道府県指定、地点種別（官署のみ／アメダス含む）。

#### (b) 地点別ランキング
- 特定の1地点を選び、その地点における「観測史上の最高気温トップN」を表示。
- 地点ページ（4.4節の一部）に埋め込まれる形で提供。
- 自己ベスト・自己ワースト（＝最も暑かった日々）の推移が一覧できる。

#### (c) 日付別ランキング
- 特定の1日（または「8月17日」のように月日のみを指定した過去全年の比較）を選び、その日の全国観測地点を気温順にランキング。
- 「日付別詳細ランキング」として4.3節の中核機能。

#### (d) ニッチ集計ランキング（マニア向け派生指標）

`daily_max` を素材に、閾値カウント・連続記録・年別集計といった**派生指標**のランキングを提供する。これらは事前計算に頼らず、原則ブラウザ側のSQL（`sql.js-httpvfs`）で都度算出できることを設計要件とする（5.3節・6章）。想定する初期メニュー:

- **地点別 猛暑日(≥35℃)回数ランキング** / **酷暑日(≥40℃)回数ランキング**（歴代累計）。
- **最長連続猛暑日** 記録（ウィンドウ関数による連続日数の算出）。
- **年別・全国の猛暑日／酷暑日 観測地点数**（「暑かった年」ランキング）。
- **シーズン最速で35℃／40℃に到達した日** ランキング。
- **同一地点での40℃到達回数**、月別・地点別の最高気温分布 など。

新しいニッチ企画の追加が「SQLを1本書く」だけで完結することを重視する（データ取込パイプラインの改修を不要にする）。

### 4.3 日付別詳細ランキング＋その日の全国分布

- 日付を1つ選択すると:
  - その日の全国観測地点を最高気温順にランキング表示（上位N件、および都道府県ごとの最高値サマリー）。
  - 同じ画面に地図（4.4節のヒートマップ）を並べ、ランキングの行にホバー／クリックすると地図上の該当地点がハイライトされる（相互連携）。
  - その日の全国最高気温・全国の猛暑日／酷暑日の観測地点数などのサマリー統計を上部に表示。
- URLは日付ベースで一意に決まる（例: `/date/2024-08-05`）ことを想定し、SNS等での共有を意識する。

### 4.4 地図ヒートマップ

- MapLibre GL（もしくはD3＋日本のtopojson）を用いて、観測地点を緯度経度でプロット。
- 各地点のマーカーは、選択中の日付（または表示モード）における最高気温に応じて色分け（3章のカテゴリカラーをベースにした連続グラデーション、またはカテゴリ4色の離散配色を切り替え可能）。
- マーカークリックで地点ページ（4.4節下記）へ遷移。
- 表示モード切り替え:
  - 「特定日モード」: 4.3節と連携し、その日の全国分布を表示。
  - 「歴代最高モード」: 各地点の観測史上最高気温で色分け（地点ごとの記録がいつ出たかはツールチップで表示）。
- ズーム・パン対応。都道府県境界線を薄く表示し地理的な文脈を持たせる。

#### 地点ページ
- 地点基本情報（名称、都道府県、種別、緯度経度、標高、観測開始日）。
- その地点の観測史上トップN（4.2(b)のランキング埋め込み）。
- 気温の年別推移グラフ（各年の最高気温、猛暑日／酷暑日の日数推移など）。

### 4.5 デザイン: 気温連動の演出

- 3章の4カテゴリに対応し、UIのテーマが段階的にエスカレーションする。
  - 夏日: ベース配色に薄い黄色のアクセント。
  - 真夏日: 橙色のアクセントを強める。カード背景にグラデーション。
  - 猛暑日: 赤系配色＋Canvas/WebGLによる**陽炎（heat-haze）シェーダー**をページ背景または該当カードに適用（屈折風の歪みエフェクト、下から上へ立ち上る揺らめき）。
  - 酷暑日: 深紅〜マゼンタの配色＋陽炎エフェクト強化＋画面全体の微振動（screen shake）＋気温数値のパルス点滅アニメーション。
- エフェクトは「その日／その地点が到達した最高カテゴリ」に応じて自動で切り替わり、ユーザー操作を必要としない。
- パフォーマンスへの配慮: 一覧・ランキングページでは軽量なCSSベースの演出に留め、陽炎シェーダーのようなWebGL演出は「日付詳細ページ」「地点ページ」などフォーカスされた1画面でのみ有効化する。
- アクセシビリティ: `prefers-reduced-motion` を尊重し、モーション低減設定時は揺らぎ・パルス系のアニメーションを無効化しカラー変化のみ残す。

## 5. データモデル

### 5.1 テーブル定義（DDL相当）

```sql
-- 観測地点マスタ
CREATE TABLE stations (
    id             SERIAL PRIMARY KEY,
    jma_code       VARCHAR(10)  NOT NULL UNIQUE,   -- obsdl の stid（例 's47662'）。過去データDL・ランキングURLにそのまま使えるため主キーに採用（7.4節）
    name           VARCHAR(50)  NOT NULL,          -- 地点名（例: 熊谷）
    name_kana      VARCHAR(100),                   -- 検索用かな表記（例: くまがや）
    pref           VARCHAR(20)  NOT NULL,          -- 都道府県名
    lat            DECIMAL(9,6) NOT NULL,          -- 緯度
    lon            DECIMAL(9,6) NOT NULL,          -- 経度
    elevation      DECIMAL(6,1),                   -- 標高(m)
    type           VARCHAR(20)  NOT NULL,          -- 'observatory'(官署) / 'amedas'(アメダス)
    obs_start      DATE,                           -- 観測開始日
    created_at     TIMESTAMP    NOT NULL DEFAULT now(),
    updated_at     TIMESTAMP    NOT NULL DEFAULT now()
);

-- 日別最高気温
CREATE TABLE daily_max (
    id             BIGSERIAL PRIMARY KEY,
    station_id     INTEGER      NOT NULL REFERENCES stations(id),
    date           DATE         NOT NULL,
    max_temp       DECIMAL(4,1) NOT NULL,          -- 例: 41.1
    max_temp_time  TIME,                           -- 最高気温を記録した時刻（不明な場合NULL）
    quality_flag   VARCHAR(10),                    -- 品質フラグ（欠測・準正常値等、気象庁の値をそのまま保持）
    created_at     TIMESTAMP    NOT NULL DEFAULT now(),
    UNIQUE (station_id, date)
);

CREATE INDEX idx_daily_max_date       ON daily_max (date);
CREATE INDEX idx_daily_max_temp       ON daily_max (max_temp DESC);
CREATE INDEX idx_daily_max_station    ON daily_max (station_id);
```

補足:
- `quality_flag` は気象庁の品質情報（欠測、資料不足等）をそのまま保持し、ランキング計算時に除外条件として利用する。
- `max_temp_time`（最高気温の起時）は官署では取得できるがアメダスでは提供されない場合があるためNULL許容。
- 将来、最低気温など他要素を追加する場合は `daily_max` を `daily_observation` に一般化するか、別テーブルを追加するかは9章の未決事項とする。

### 5.2 3ランキングのクエリ例

#### (a) 歴代全国ランキング（上位100件、タイ順位対応）

```sql
SELECT
    RANK() OVER (ORDER BY dm.max_temp DESC) AS rank,
    s.name, s.pref, dm.date, dm.max_temp
FROM daily_max dm
JOIN stations s ON s.id = dm.station_id
WHERE dm.quality_flag IS NULL OR dm.quality_flag = 'normal'
ORDER BY dm.max_temp DESC
LIMIT 100;
```

#### (b) 地点別ランキング（指定地点の観測史上トップ20）

```sql
SELECT
    RANK() OVER (ORDER BY dm.max_temp DESC) AS rank,
    dm.date, dm.max_temp
FROM daily_max dm
WHERE dm.station_id = :station_id
ORDER BY dm.max_temp DESC
LIMIT 20;
```

#### (c) 日付別ランキング（指定日の全国ランキング）

```sql
SELECT
    RANK() OVER (ORDER BY dm.max_temp DESC) AS rank,
    s.name, s.pref, s.lat, s.lon, dm.max_temp
FROM daily_max dm
JOIN stations s ON s.id = dm.station_id
WHERE dm.date = :target_date
ORDER BY dm.max_temp DESC;
```

#### 補足: 「月日一致」ランキング（例: 過去すべての8月17日の比較）

```sql
SELECT
    RANK() OVER (ORDER BY dm.max_temp DESC) AS rank,
    s.name, s.pref, dm.date, dm.max_temp
FROM daily_max dm
JOIN stations s ON s.id = dm.station_id
WHERE EXTRACT(MONTH FROM dm.date) = :month
  AND EXTRACT(DAY   FROM dm.date) = :day
ORDER BY dm.max_temp DESC
LIMIT 100;
```

> **SQLダイアレクトについて**: 上記のDDL/クエリはPostgres表記で概念を示している。実装の正準ストアは **SQLite** のため、`SERIAL`→`INTEGER PRIMARY KEY AUTOINCREMENT`、`EXTRACT(MONTH FROM date)`→`strftime('%m', date)`、`now()`→`CURRENT_TIMESTAMP` 等に読み替える。`RANK() OVER (...)` などのウィンドウ関数はSQLite（および`sql.js`）でそのまま利用できる。

### 5.3 配信・パフォーマンス方針（SQLite主軸＋事前計算JSON）

Postgresのマテリアライズドビューは持たない。GitHub Actionsのビルド時に生成する**成果物**として、以下の2系統を静的配信する。

- **正準ストア: 単一の SQLite ファイル**（`data/max_temp.sqlite`）
  - `stations` と `daily_max` の全量を格納し、5.1節のインデックスを付与する。
  - ブラウザ側は `sql.js-httpvfs` でこのファイルをマウントし、**HTTP Rangeリクエストで必要なDBページのみを取得**してSQLを実行する。全量DLは不要。
  - これにより、日付別ランキング・地点別ランキング・4.2(d)のニッチ集計まで、**サーバーなしでフロントの任意クエリ**が成立する。ビルド時に `ANALYZE` を実行し、適切なページサイズ（例: 4KB/8KB）でインデックスを効かせて転送量を抑える。
- **高速化のための事前計算JSON**（定番の重い集計・初期表示分のみ）
  - `data/ranking/national_top1000.json`: 歴代全国ランキング上位1000件。
  - `data/station/{jma_code}.json`: 地点ごとの基本情報＋自己ベストTOP・年別サマリー。
  - `data/daily/{YYYY-MM-DD}.json`: 日付ごとの全国分布（地図＋ランキング）。クリックした日だけ遅延ロードする。
  - `data/stations.json`: 地点マスタ一覧（地図の初期プロット・検索用インデックス）。
- 使い分け: **初期表示・SNS共有される固定URL（トップ／日付／地点）は事前JSONで即表示**、**ユーザーが自由に組み立てるニッチ集計・絞り込みはSQLiteに委譲**する二層構成。
- データ量の目安: 官署約150地点×全期間で `daily_max` は数百万行規模。SQLiteファイルは概ね数十〜百MB程度を見込むが、httpvfsの部分取得により初期転送は数百KB〜数MBに収まる想定。インデックス設計は5.1節に準拠し、実測しながら調整する。

## 6. 技術構成（表）

**アーキテクチャ全体像**: サーバーレス（ランタイムDBサーバーなし）の**静的サイト＋ビルド成果物としてのDB**構成。GitHub Actionsが気象庁からデータを取り込み、SQLiteと事前計算JSONをビルドしてGitHub Pagesへデプロイする。ブラウザは静的ファイルを読み、必要に応じて `sql.js-httpvfs` で任意SQLを実行する。

| レイヤー | 採用技術 | 備考 |
|---|---|---|
| フロントエンド | React + TypeScript（静的書き出し） | Next.js `output: export` もしくは Vite SPA。SSRサーバーは持たず全ページを静的生成 |
| スタイリング | Tailwind CSS | ユーティリティベース。カテゴリ別テーマをCSS変数で切替 |
| 演出エフェクト | Canvas API / WebGL（陽炎シェーダー） | 猛暑日以上のページ・カードに限定適用（4.5節参照） |
| 地図 | MapLibre GL JS（第一候補）／D3 + 日本 topojson（代替） | ベクトルタイルでズーム・パンに対応 |
| データ配信・クエリ | **SQLite + `sql.js-httpvfs`（ブラウザ内SQL）** | HTTP Rangeで部分取得。ニッチ集計をフロントSQLで実行（5.3節） |
| 事前計算 | 静的JSON（ビルド時生成） | トップ／日付／地点の初期表示・SNS共有URL用（5.3節） |
| ビルド用DB / ETL | SQLite ＋ Python (pandas) | 取込・正規化してSQLiteを構築。配信ファイルもここで生成 |
| データ取込 | Python (pandas) | 気象庁データの取得／CSV変換／クレンジング／ロード |
| 自動更新 | **GitHub Actions（cron）** | 日次で取込→SQLite/JSONビルド→Pagesデプロイ |
| ホスティング | **GitHub Pages** | 静的配信。ランタイムDB・APIサーバー不要・無料 |

> Prismaは不要（ランタイムDB接続がないため）。スキーマは5.1節のDDL（SQLite方言）を正準とし、ETLスクリプトが直接管理する。将来リアルタイム性やユーザー投稿が必要になった場合のみ、外部Postgres＋API（Vercel/Supabase等）への移行を9章の検討事項として残す。

## 7. データ取込の方針とリスク

### 7.1 気象庁データの特性

気象庁は汎用のREST APIを公開していない。データ取得は主に以下の3系統のWebインターフェース／ダウンロード機構に依存する。

1. **歴代全国ランキング**（歴代全国ランキング/観測史上1〜10位等のページ）: 気象庁サイトの「歴代全国ランキング」ページから、既に集計済みの上位記録を取得できる。地点網羅性は高いが、あくまで「上位のみ」であり全量データではない。
2. **観測地点マスタ**: 気象庁の観測地点一覧（官署・アメダスの地点名・地点番号・緯度経度・標高等）のページ／ダウンロードデータから取得。地点の統廃合・改称履歴があるため、地点コードの継続性に注意が必要。
3. **過去の気象データ検索・ダウンロード（いわゆる `obsdl` エンドポイント）**: 地点・期間・要素を指定してCSV相当のデータを取得できる、気象庁の「過去の気象データ・ダウンロード」機能。日別値の全量取得にはこれが本命だが、非公式な内部エンドポイントであり、一度に取得できる地点数・期間・要素数に制限がある（多地点×長期間の一括取得は不可、分割リクエストが必要）。

### 7.2 リスクと対応方針

| リスク | 内容 | 対応方針 |
|---|---|---|
| API仕様変更 | `obsdl` 等は非公式運用のため、URL構造やレスポンス形式が予告なく変わる可能性 | 取込スクリプトをモジュール化し、パーサー部分を差し替えやすくする。取込ログとレスポンスのスナップショットを保存し、異常検知しやすくする |
| 取得負荷・地点数制限 | 一括取得の地点数・期間に上限があるため、全アメダス(約1300地点)×全期間の取得は現実的でない | まず**官署（気象台・測候所等、全国約150地点）の全期間**取込を優先し、アメダス地点は後続フェーズで段階的に拡張する（8章参照） |
| データ品質・欠測 | 観測機器更新、地点移転、戦中戦後の欠測等が存在 | `quality_flag` で保持し、ランキング計算時にフィルタ。地点移転履歴があるものは注記をUIに出す（将来検討） |
| 利用条件・出典明記 | 気象庁データの二次利用には出典の明記が利用条件として求められる | 全ページのフッター、および個別データ表示箇所に「出典: 気象庁」を明記。加工・解析している旨も明示し、公式発表値との差異が生じた場合は気象庁公式を正とする免責を記載 |
| サーバー負荷への配慮 | スクレイピング的な取得は気象庁サーバーへの負荷になり得る | リクエスト間隔を空ける（レートリミット）、キャッシュ・再取得の要否を判断してから実行、日次バッチとして深夜帯に実行 |

### 7.4 実測で確定した事実（2026-07-22 フィージビリティ調査）

`ingest/FINDINGS.md` に基づく、実際に取得・検証済みの確定事項。設計はこの実測に従う。

- **ネットワーク到達性 ○**: 3系統すべて実データ取得済み。歴代ランキング（GET・セッション不要）、地点マスタ、obsdl（POST・CSV）すべて動作確認済み。
- **地点IDが3体系並存（最重要の落とし穴）**: 同一地点に (a)`amedastable.json` の数値コード、(b)obsdl の `stid`（例 `s47662`）、(c)etrn の `prec_no`+`block_no` が並存し**相互に一致しない**。`stid` から英字接頭辞を除いた数値が etrn の `block_no` と一致する規則性のみ確認。**座標・かなは amedastable、DL/ランキングは stid が必要**なため、地点マスタは**2ソースを地点名でJOIN**して生成する（`fetch_stations.py` で実装済み・全国1723行・約20秒で完走）。同名衝突が約38件あり手動QAが必要。
- **`type` フィールドで気温有無を判定してはいけない**: `type=C`（公称「雨量のみ」）でも実際には最高気温が取れる地点がある（野沢温泉で実証）。気温データの有無は**実際に叩いて確認**するのが唯一信頼できる方法。
- **obsdl 要素コード `202` = 日別最高気温**（`201`平均/`203`最低/`204`月間平均最高 と紛らわしいので注意）。**1地点なら全履歴を1リクエストで取得可**（東京35年12,784行を1回で取得実証）。リクエスト上限は概算 `地点数×要素数×日数 ≤ 44000`。
- **CSVエンコーディングは cp932**（UTF-8ではない）。品質情報・均質番号列あり → `quality_flag` にマップ。**最高気温の起時（`max_temp_time`）は要素202には含まれない**（別要素の追加調査が必要・未実施）。
- **`obs_start` の実データソース**: 地点別ランキング `rank_s.php` が「統計期間」を返す（地点ごと1リクエスト）。
- **現在の歴代1位は 群馬県伊勢崎 41.8℃（2025-08-05）**。2位タイに静岡41.4/埼玉鳩山41.4（ともに2025）。**2018/2020の41.1（浜松・熊谷）は既に更新済み**——2025年の猛暑で塗り替わっている。モックの数字は要更新。
- **ライセンス**: 気象庁は「公共データ利用規約（第1.0版）」準拠。**出典明記を条件に自由利用可**、加工した旨も明記。今回の用途は気象業務法17/23条に抵触しない。

### 7.3 取込アーキテクチャ概要（GitHub Actions パイプライン）

```
[GitHub Actions cron（日次）]
      │
      ├─ 1. fetch: 気象庁の3系統から取得（地点マスタ / 歴代ランキング / obsdl日別値）
      ├─ 2. normalize: pandas でクレンジング・正規化（CSV中間ファイル）
      ├─ 3. load: SQLite (data/max_temp.sqlite) を構築、インデックス付与、ANALYZE
      ├─ 4. build views: 事前計算JSON（national_top1000 / daily / station / stations）を生成
      └─ 5. deploy: 静的サイト＋data/ を GitHub Pages へデプロイ
```

- Python (pandas) のバッチスクリプトで取得・正規化し、**SQLiteを正準ストア**として構築。同一ジョブ内で配信用の事前計算JSON（5.3節）も出力する。
- 地点マスタの更新は低頻度（地点の新設・廃止時のみ）、日別値の取込は日次バッチとして運用する。差分取込（前回取得日以降のみ追記）で気象庁サーバー負荷とビルド時間を抑える。
- 取込スクリプトとフロントエンドは疎結合とし、**5.1節のDDL（SQLite方言）を共通の契約**とする（ORMは介さない）。
- 巨大なSQLite/データをGit本体に載せる是非（Git LFS利用 or Actionsのアーティファクト経由でPagesへ直接デプロイしリポジトリにはコミットしない）は運用で判断する（9章）。

## 9. 未決事項 / オープンな問い

- ~~**ホスティング先の確定**~~: **確定済み（v0.2）**。GitHub Pages + Actions、SQLite(`sql.js-httpvfs`)主軸＋事前計算JSON。将来リアルタイム性／ユーザー投稿が必要になった場合のみ外部Postgres＋APIへの移行を再検討する。
- **大容量SQLite/データの配置**: 数十〜百MBのSQLiteをGit本体にコミットするか、Git LFS or Actionsアーティファクト経由でPagesへ直接デプロイ（リポジトリ非コミット）とするか。Pagesの容量・帯域制限との兼ね合いで決定する。
- **アメダス全地点（約1300）の取込範囲**: 全地点×全期間は現実的でないため、優先順位（都道府県庁所在地優先／記録更新頻度の高い地点優先等）の基準を今後決める必要がある。
- **地点IDの3体系JOIN（一部確定）**: `jma_code` は obsdl の `stid` を採用済み（7.4節）。ただし座標取得元の `amedastable.json` とは共通キーが無く**地点名JOIN**が必要で、同名衝突が約38件残る。座標突合等での自動解消 or 手動QAリストの整備が課題。地点の統廃合・改称（末尾 `*` 脚注）の扱いも継続課題。
- **最高気温の起時（`max_temp_time`）の取得**: 要素202には時刻が含まれない。時別値等の別要素で取得できるか要追加調査（現状NULL許容で運用可）。
- **他気象要素（最低気温・降水量等）への拡張可否**: 将来のスコープ拡張の是非とデータモデルへの影響（`daily_max` の一般化 vs 別テーブル化）。
- **多言語対応（英語UI）の要否**: 海外の気象ファンからの需要があるか未検証。
- **陽炎シェーダーの実装方式**: 独自WebGLシェーダー実装か、既存ライブラリ（例: pixi.js フィルタ等）の活用かは技術検証が必要。
- **キャッシュ／ISR戦略の詳細**: 日付詳細ページ・地点ページの再検証間隔（revalidate）や、マテリアライズドビューのリフレッシュタイミングとの整合。
- **利用規約・出典表記の具体文言**: 気象庁の利用規約を精読の上、フッター表記の最終文言を確定させる必要がある。
- **同着（タイ記録）のUI表現**: ランキングで同一気温が並ぶ場合の視覚的な扱い（同順位バッジの出し方等）。
